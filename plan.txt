## plan.txt — hierarchical-agentic-qa (Lab4 → Lab5 extension)
# PURPOSE: Build a clean, resume-grade project that demonstrates
# hierarchical agent orchestration, model-per-role benchmarking, and
# measurable evaluation signals. Be explicit and avoid assumptions.

---

## 0) Target outcome (one sentence)
A hierarchical agentic QA system where a Planner emits a JSON plan, a Researcher executes tool calls (PDF RAG + Tavily web search), and an Answerer synthesizes the final output, with per-role model benchmarking and structured JSONL logs that prove which model ran where and how it behaved.

---

## 1) Non-negotiable constraints
1) Readability over cleverness: explicit code, simple flow, clear logs.
2) CPU-only: FAISS CPU + embeddings on CPU.
3) All models are local Ollama models accessed via LangChain.
4) Logs must clearly show which model was used at each role and step.
5) Evaluation uses 10 fixed questions with pass/fail checks.
6) One-role-at-a-time model comparisons (no full combinatorial grid).
7) LangSmith tracing is OPTIONAL and must be behind a toggle.
8) Redaction MUST be applied before ANY logging or tracing.
9) Rebuild everything in the new project (PDF index + artifacts).
10) Use conda env: ollama.

## 4) One-role-at-a-time benchmarking plan
We vary ONE role at a time while keeping the other two fixed.

Planner sweep:
- deepseek-r1:1.5b → qwen3:1.7b → ministral-3:3b
(Researcher: qwen3:1.7b, Answerer: llama3.2:3b fixed)

Researcher sweep:
- qwen3:0.6b → qwen3:1.7b
(Planner: deepseek-r1:1.5b, Answerer: llama3.2:3b fixed)

Answerer sweep:
- llama3.2:3b → ministral-3:3b
(Planner: deepseek-r1:1.5b, Researcher: qwen3:1.7b fixed)

The eval runner must:
- iterate through each sweep preset,
- run the 10 questions,
- log timing + pass/fail metrics.

## 5) Planner responsibilities (explicit)
Planner MUST:
- Read the user query + conversation state
- Emit VALID JSON with a list of steps (3–6 steps)
- Include explicit tool steps (pdf/web/both) and reasoning steps
- Restrict actions to: "researcher", "answerer"
- Store plan in state.plan and advance step_index

JSON output format (strict):
{
  "steps": [
    {"id": 0, "action": "researcher", "description": "...", "tools": ["retrieve_context"]},
    {"id": 1, "action": "researcher", "description": "...", "tools": ["tavily_search"]},
    {"id": 2, "action": "answerer", "description": "Synthesize answer with sources"}
  ]
}

If JSON parsing fails, fallback to a safe default plan (researcher → answerer).

---

## 6) Researcher responsibilities (explicit)
Researcher MUST:
- Use tools based on the Planner plan
- Call at least one tool
- Return structured notes + citations
- Tag each note with its source (pdf page or web URL)
- Never produce final answer text

Tools:
- retrieve_context(query, k)
- tavily_search(query, max_results)

---

## 7) Answerer responsibilities (explicit)
Answerer MUST:
- Use researcher notes
- Output final answer in clean prose
- Include a "Sources" section:
  - PDF sources as [p.X]
  - Web sources as URLs

---

## 8) Logging & evaluation (explicit)
### 8.1 JSONL log format (single file per run)
Each step append one JSON object with:
- run_id
- step_index
- role
- model
- action
- tool_calls
- tool_results (redacted + truncated)
- latency_ms
- guardrail_checks (pass/fail)
- errors (if any)

### 8.2 Guardrail checks (pass/fail)
Required:
1) JSON plan parseable
2) Tool called before researcher response
3) Correct tool choice based on query type:
   - PDF if question is about Deep Work
   - Web if current events / recent info
   - Both if explicitly mixed
4) Citations included when PDF used
5) Web sources included when web used

---

## 9) Redaction rules (explicit)
Before logging or tracing:
- Mask API keys, tokens, emails, secrets
- Truncate tool outputs to 500 chars
- Redaction MUST happen before LangSmith tracing too
